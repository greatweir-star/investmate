# 后端架构

## 1. 技术选型

MVP 后端建议使用 FastAPI。

原因：

- Python 生态适合数据处理；
- 易于接入 Tushare；
- 易于接入 AI 服务；
- API 文档自动生成；
- 适合快速迭代。

## 3. 核心模块

### 3.1 config

读取环境变量。

字段：

- DATABASE_URL；
- TUSHARE_TOKEN；
- OPENAI_API_KEY；
- OPENAI_MODEL。

### 3.2 database

管理数据库连接。

MVP 可以先使用 SQLAlchemy。

### 3.3 tushare_service

负责数据拉取。

职责：

- 拉取标的基础信息；
- 拉取日线行情；
- 拉取指数数据；
- 拉取每日基本面；
- 记录同步状态。

### 3.4 signal_service

负责基础信号计算。

信号：

- 趋势；
- 波动；
- 流动性；
- 估值；
- 用户匹配度。

### 3.5 analysis_service

负责组合结构化结果。

输入：

- 标的数据；
- 用户画像；
- 市场状态；
- 持仓数据。

输出：

- 结构化分析对象。

### 3.6 ai_service

负责将结构化结果转为中文解释。

注意：AI 服务不直接决定核心评分，评分应来自规则和数据。

## 4. API 模块

### market.py

接口：

- GET `/api/market/status`

### assets.py

接口：

- GET `/api/assets/search`
- POST `/api/analysis/asset`

### portfolio.py

接口：

- POST `/api/portfolio`
- POST `/api/analysis/portfolio`

### history.py

接口：

- GET `/api/analysis/history`
- GET `/api/analysis/{analysis_id}`

### settings.py

接口：

- GET `/api/settings`
- POST `/api/settings`

## 5. 数据处理原则

1. 数据层和 AI 层分离。
2. 评分和规则优先由系统计算。
3. AI 主要负责解释和表达。
4. 每次分析都要保存输入与输出。
5. 数据缺失时返回明确错误，不生成伪完整结果。

## 6. MVP 返回格式

所有 API 返回统一结构：

```json
{
  "success": true,
  "data": {},
  "error": null,
  "meta": {
    "version": "v0.1"
  }
}
```

## 7. 开发优先级

第一批：

1. main.py；
2. health 接口；
3. market status mock；
4. asset analysis mock；
5. portfolio analysis mock；
6. history mock。

第二批：

1. 数据库连接；
2. Tushare 接入；
3. 信号计算；
4. AI 服务接入。
