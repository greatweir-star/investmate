# 数据同步设计

## 1. 目标

数据同步模块负责把外部数据源中的基础数据同步到本地数据库。

MVP 阶段只接入 Tushare，并且只做日频数据，不做盘中实时能力。

## 2. 同步范围

第一阶段同步以下数据：

1. 标的基础信息；
2. 日线行情；
3. 指数行情；
4. 每日基本面；
5. 交易日历。

## 3. 同步方式

### 3.1 手动同步

设置页提供同步按钮。

适合开发阶段和测试阶段。

### 3.2 定时同步

后续可增加定时任务。

建议时间：交易日收盘后。

## 4. 同步流程

```text
读取配置
  ↓
检查数据源 Token
  ↓
判断同步范围
  ↓
调用数据源 API
  ↓
写入原始数据表
  ↓
清洗并写入标准化数据表
  ↓
计算基础信号
  ↓
记录同步日志
```

## 5. 数据状态

系统需要记录：

- 最新同步日期；
- 同步数据类型；
- 同步状态；
- 行数；
- 错误信息；
- 开始时间；
- 结束时间。

## 6. API 设计

### 6.1 触发同步

```http
POST /api/admin/sync
```

请求：

```json
{
  "scope": ["asset_basic", "daily_price", "index_daily"],
  "trade_date": "2026-06-28"
}
```

### 6.2 获取同步状态

```http
GET /api/admin/sync/status
```

返回：

```json
{
  "latest_data_date": "2026-06-28",
  "status": "success",
  "items": [
    {
      "data_type": "daily_price",
      "row_count": 5300,
      "status": "success"
    }
  ]
}
```

## 7. 错误处理

需要处理：

- Token 缺失；
- Token 无效；
- 网络错误；
- 数据为空；
- 字段缺失；
- 重复数据；
- 单次请求数量限制。

错误信息需要写入同步日志。

## 8. 数据质量检查

每次同步后检查：

1. 核心指数是否存在；
2. 行情数据行数是否明显异常；
3. 关键字段是否为空；
4. 数据日期是否符合预期；
5. 是否存在重复记录。

## 9. MVP 实现建议

第一版可以先实现同步接口和 mock 状态，再接入真实 Tushare。

优先级：

1. 标的基础信息；
2. 指数行情；
3. 日线行情；
4. 每日基本面；
5. 交易日历。

## 10. 验收标准

完成后应支持：

- 在设置页触发数据同步；
- 在后端查看同步日志；
- 分析接口可以读取最新数据日期；
- 数据异常时前端能够提示。
