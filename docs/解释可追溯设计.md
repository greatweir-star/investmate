# 解释可追溯设计

## 1. 目标

解释可追溯设计用于确保 InvestMate 的每个重要解释都能回到结构化依据。

用户看到的不是凭空生成的自然语言，而是基于输入、数据、规则和历史上下文生成的说明。

## 2. 可追溯对象

需要可追溯的对象包括：

- 组合健康度；
- 分项评分；
- 分布结论；
- 关注事项；
- 中文解释；
- 复盘变化；
- 数据不足提示。

## 3. 解释结构

每个重要解释建议包含三层：

```text
结论
  ↓
依据
  ↓
限制或注意事项
```

示例结构：

```text
结论：组合集中度需要关注。
依据：前三大持仓占比较高。
注意：当前只基于用户录入比例和基础分类判断。
```

## 4. Evidence 字段

报告字段中应保留 evidence。

建议字段：

- evidence_id；
- evidence_type；
- source_field；
- source_value；
- description。

## 5. 页面展示

页面不一定默认展开所有依据，但需要允许用户查看：

- 为什么有这个评分；
- 为什么出现这个关注事项；
- 使用了哪些输入字段；
- 哪些数据缺失；
- 生成时间和数据日期。

## 6. Agent 要求

报告解释 Agent 必须基于 structured_report。

禁止：

- 脱离输入生成新判断；
- 忽略数据缺失；
- 不说明前提；
- 生成无法追溯的解释。

## 7. 历史追溯

历史记录需要保存：

- 用户输入；
- 结构化结果；
- evidence；
- 中文解释；
- Prompt 版本；
- 模型版本；
- 数据日期。

## 8. MVP 实现

MVP 阶段至少做到：

- 每个 score_item 有 explanation；
- 每个 watch_item 有 evidence；
- 报告展示数据日期；
- 历史保存 structured_result。

## 9. 验收标准

- 用户可以查看评分依据；
- 用户可以查看关注事项依据；
- 历史记录保留结构化结果；
- 解释不脱离结构化结果；
- 数据不足时有明确说明。
