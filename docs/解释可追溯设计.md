# 解释可追溯设计

## 1. 目标

解释可追溯设计用于确保 InvestMate 的每个重要解释都能回到结构化依据。

用户看到的不是凭空生成的自然语言，而是基于输入、数据、规则和历史上下文生成的说明。

## 2. 可追溯对象

需要可追溯的对象包括：

- 组合健康度；
- 分项评分；
- 分布结论；
- 关注事项；
- 中文解释；
- 复盘变化；
- 数据不足提示。

## 3. 追溯链路

每个重要结论都应能追溯到：

```text
用户输入
  ↓
数据补全
  ↓
规则计算
  ↓
结构化字段
  ↓
AI 解释
  ↓
页面展示
```

## 4. Evidence 字段

建议每个 ScoreItem 和 WatchItem 都包含 evidence 字段。

字段结构：

- source_type；
- source_field；
- source_value；
- description。

示例：

```text
source_type: position
source_field: weight
source_value: 0.45
description: 最大单一持仓比例较高，因此集中度得分降低。
```

## 5. 页面展示方式

页面不需要默认展示全部依据，但需要支持展开。

建议展示：

- 为什么这么说；
- 依据哪些字段；
- 数据日期；
- 哪些数据缺失；
- 这个解释的限制。

## 6. AI 解释约束

AI 解释必须引用已有结构化字段。

不允许出现：

- 结构化结果中不存在的事实；
- 没有依据的判断；
- 绝对化表述；
- 未说明数据来源的结论。

## 7. 历史追溯

历史报告需要保存：

- 当时的输入；
- 当时的数据日期；
- 当时的评分；
- 当时的解释；
- 当时的 Prompt 版本；
- 当时的模型版本。

这样用户回看时，不会把后续数据误认为当时依据。

## 8. 复盘追溯

周度复盘中的变化说明必须能追溯到两个报告：

- previous_report_id；
- current_report_id。

变化项需要说明：

- 上次值；
- 本次值；
- 变化方向；
- 解释说明。

## 9. 后端要求

后端生成报告时需要保留中间结果：

- enriched_positions；
- score_calculation；
- exposure_calculation；
- watch_item_evidence；
- agent_output；
- boundary_check_result。

MVP 可以先保存在 JSONB 中。

## 10. 验收标准

- 重要结论都有 evidence；
- 页面可以展示依据；
- 历史报告可以回看当时依据；
- 复盘变化可以追溯到前后报告；
- AI 解释不脱离结构化结果。
