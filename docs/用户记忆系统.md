# 用户记忆系统

## 1. 目标

用户记忆系统用于让 InvestMate 从一次性报告工具，逐步演进为长期复盘产品。

记忆不是为了替用户做决定，而是为了让后续报告更贴近用户的历史、偏好和理解方式。

## 2. 记忆类型

第一阶段建议记录五类记忆：

1. 用户投资 DNA；
2. 历史报告；
3. 用户反馈；
4. 历史关注事项；
5. 解释偏好。

## 3. 用户投资 DNA

记录用户相对稳定的偏好和约束。

字段建议：

- risk_level；
- investment_horizon；
- asset_preferences；
- experience_level；
- max_drawdown_tolerance；
- explanation_style。

用途：

- 影响报告中的用户匹配度；
- 影响解释深度；
- 影响周度复盘关注点。

## 4. 历史报告记忆

每次报告都应保存：

- report_id；
- report_type；
- input_positions；
- score_items；
- exposure_items；
- watch_items；
- explanation；
- data_date；
- generated_at。

用途：

- 支持历史回看；
- 支持周度复盘对比；
- 支持关注事项延续。

## 5. 用户反馈记忆

记录用户对报告的反馈。

字段建议：

- feedback_type；
- feedback_text；
- target_module；
- created_at。

反馈类型：

- 有帮助；
- 看不懂；
- 不准确；
- 想看更多；
- 不需要。

用途：

- 优化解释；
- 调整模块顺序；
- 识别高意向用户；
- 支持产品迭代。

## 6. 历史关注事项

报告中的 watch_items 不应只展示一次。

如果某个关注事项连续出现，应在周度复盘中提示：

```text
该事项已连续多次出现在报告中。
```

用途：

- 延续上下文；
- 提高复盘价值；
- 帮用户识别重复出现的问题。

## 7. 解释偏好

系统可以记录用户更喜欢的解释方式：

- 简洁；
- 详细；
- 更白话；
- 更数据化；
- 更偏复盘。

MVP 阶段可以通过反馈人工判断，后续再自动更新。

## 8. 记忆更新规则

### 8.1 自动更新

- 新报告生成后，写入历史报告；
- 用户提交反馈后，写入反馈记忆；
- watch_items 写入关注事项历史。

### 8.2 用户确认后更新

- 投资 DNA；
- 解释偏好；
- 重要用户标签。

### 8.3 不自动更新

不要根据一次行为直接改变用户长期画像。

## 9. 隐私和边界

- 用户应知道哪些信息会被保存；
- 用户应可以清除历史记录；
- 用户应可以修改投资 DNA；
- 敏感字段不应明文展示。

## 10. 数据表建议

可在后续数据库设计中增加：

- user_memory_profile；
- user_feedback_memory；
- user_watch_item_history；
- user_explanation_preference。

MVP 可以先用 JSONB 存储，后续再拆表。

## 11. 验收标准

- 报告可以进入历史记忆；
- 用户反馈可以被记录；
- 周度复盘可以读取上次报告；
- 用户偏好可以影响解释；
- 用户可以修改或清除个人配置。
